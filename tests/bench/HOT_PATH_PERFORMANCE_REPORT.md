# Отчет по производительности горячего пути Lite FSM

Дата замера: 2026-04-28  
Цель: получить воспроизводимую базовую оценку стоимости одного игрового события `TICK` в `MachineManager.transition()` для дальнейшего анализа горячего пути и оптимизаций.

## Краткий вывод

Текущая оценка в production-режиме показывает, что основной лимит кадра задают не доменные машины, а число акторов, до которых доставляется событие.

- Для события без области адресации (`unscoped`), которое проходит через все доменные машины и всех подходящих акторов, практический безопасный диапазон для `16ms` кадра находится около `5k` активных акторов при `100` доменных машинах.
- `6k` акторов уже пограничный сценарий: `p95` еще около `15.5ms`, но `p99` выходит за кадр.
- `7k+` акторов в `unscoped`-режиме дают стабильные просадки выше `16ms`.
- Адресация по `groupId` заметно дешевле и должна быть основным способом тика для игровых объектов, когда обновляется только часть мира.
- Адресация по `groupTag` помогает только если сам тег ограничивает рабочий набор. Событие по тегу на `10k` акторов остается тяжелым.
- Нагрузка только на доменные машины сравнительно дешевая: даже `10k` доменных машин в контрольном прогоне были в среднем около `6.8ms`, хотя `p95` может ловить спайки runtime/GC.

Практический вывод для игрового цикла: не стоит делать каждый `16ms` тик `unscoped`-событием на десятки тысяч акторов. Для 60fps архитектурно лучше тикать группы, чанки или видимые зоны, а `unscoped`-события оставить для редких глобальных событий.

## Что измерялось

Измерялась стоимость одного вызова:

```ts
manager.transition({ type: "TICK" });
```

или вариантов с адресацией:

```ts
manager.transition({ type: "TICK", meta: { groupTag: "enemy" } });
manager.transition({ type: "TICK", meta: { groupId: ["enemy/0", "enemy/1"] } });
```

Каждый сценарий заранее создавал `MachineManager`, доменные машины, шаблоны акторов и нужное количество уже заспавненных акторов. Время подготовки и spawn не включалось в измерение кадра. Внутри замера измерялся только steady-state `transition()` на уже подготовленном состоянии.

## Модель сценария

Доменная машина:

- Имеет одно публичное состояние `ACTIVE`.
- Обрабатывает `TICK` через переход в себя.
- Reducer возвращает новый `{ state, context }`, увеличивая счетчик.

Шаблон актора:

- Имеет `__INIT -> ALIVE` через spawn-событие.
- В `ALIVE` обрабатывает `TICK` через переход в себя.
- Reducer возвращает новый контекст актора: `ticks`, `x`, `y`.
- Группы акторов создавались через `groupTag` и `groupId`.

Основные игровые распределения:

| Сценарий | Доменные машины | Группы акторов | Акторы |
| --- | ---: | ---: | ---: |
| Граница 5k | 100 | 500 групп x 10 акторов | 5,000 |
| Граница 6k | 100 | 600 групп x 10 акторов | 6,000 |
| Граница 7k | 100 | 700 групп x 10 акторов | 7,000 |
| Граница 8k | 100 | 800 групп x 10 акторов | 8,000 |
| Граница 10k | 100 | 1,000 групп x 10 акторов | 10,000 |
| Смешанный игровой мир | 100 | враги: 1,000 x 10; снаряды: 2,500 x 1; NPC: 100 x 50 | 17,500 |
| Только доменные машины | 100-10,000 | нет | 0 |

## Методика измерения

Команды запускались из корня репозитория:

```bash
NODE_ENV=production node --loader ts-node/esm --experimental-specifier-resolution=node --input-type=module <benchmark-script>
```

Контрольный steady-state прогон для пограничных сценариев запускался с явным GC перед сбором выборки:

```bash
NODE_ENV=production node --expose-gc --loader ts-node/esm --experimental-specifier-resolution=node --input-type=module <benchmark-script>
```

Методика:

- `NODE_ENV=production`, чтобы отключить dev-only `deepFreeze`.
- Несколько прогревочных transition перед сбором выборки.
- Время измерялось через `performance.now()` вокруг одного `manager.transition()`.
- Для контрольного steady-state прогона выполнялся `global.gc()` до прогрева и перед сбором выборки.
- Считались среднее значение, `p50`, `p90`, `p95`, `p99` и максимум.
- React render, React hooks, subscribers, middleware, actor effects, browser layout/paint и игровая логика вне reducer не измерялись.
- Измерялся TypeScript source через loader, а не собранный/minified `dist`.

## Основные результаты: steady-state граница

Контрольный прогон с `--expose-gc`, `100` доменными машинами и группами акторов по `10` акторов.

| Сценарий | Среднее | p50 | p90 | p95 | p99 | Максимум | Статус 16ms |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| `Unscoped`, 5k акторов | 4.884ms | 4.110ms | 10.282ms | 11.031ms | 11.434ms | 11.434ms | В пределах |
| `groupId` scope, 1k из 5k акторов | 2.374ms | 2.146ms | 2.520ms | 3.000ms | 9.297ms | 9.305ms | В пределах |
| `Unscoped`, 6k акторов | 7.433ms | 5.876ms | 14.742ms | 15.492ms | 21.811ms | 21.811ms | Погранично |
| `groupId` scope, 1k из 6k акторов | 2.876ms | 2.470ms | 3.348ms | 3.768ms | 12.013ms | 13.401ms | В пределах |
| `Unscoped`, 7k акторов | 9.486ms | 6.957ms | 19.904ms | 22.904ms | 25.187ms | 25.187ms | Выше бюджета |
| `groupId` scope, 1k из 7k акторов | 3.397ms | 2.862ms | 3.559ms | 5.212ms | 15.333ms | 15.695ms | В пределах |
| `Unscoped`, 8k акторов | 11.316ms | 8.110ms | 23.921ms | 25.553ms | 26.006ms | 26.006ms | Выше бюджета |
| `groupId` scope, 1k из 8k акторов | 3.687ms | 3.100ms | 3.933ms | 4.910ms | 18.058ms | 18.886ms | В основном в пределах, p99-спайк |
| `Unscoped`, 10k акторов | 16.329ms | 11.007ms | 30.415ms | 33.342ms | 38.352ms | 38.352ms | Выше бюджета |
| `groupId` scope, 1k из 10k акторов | 4.749ms | 3.948ms | 5.240ms | 6.570ms | 24.954ms | 25.300ms | p95 в пределах, p99-спайк |

Интерпретация:

- `5k` акторов — текущая практическая цель для `unscoped` события в пределах кадра.
- `6k` акторов могут пройти по `p95` в контролируемом прогоне, но `p99` уже превышает бюджет кадра.
- `7k+` акторов стоит считать превышением бюджета для `unscoped` тика каждый кадр.
- `groupId` scope остается жизнеспособным, потому что число доставленных акторов ограничено выбранными группами, а не общим числом живых акторов.

## Дополнительные результаты: смешанный игровой мир

Смешанный мир: `100` доменных машин, `17,500` акторов, `3,600` групп.

| Сценарий | Среднее | p50 | p90 | p95 | Максимум | Статус 16ms |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| `Unscoped`, все 17.5k акторов | 29.901ms | 22.234ms | 44.204ms | 45.526ms | 46.687ms | Выше бюджета |
| `groupTag: enemy`, 10k акторов | 17.467ms | 12.529ms | 33.416ms | 33.771ms | 34.348ms | Выше бюджета |
| `groupTag: npc`, 5k акторов | 16.193ms | 4.925ms | 24.022ms | 25.197ms | 624.424ms | Шумный прогон / выше p95 |
| `groupId` scope, 50 enemy-групп, 500 акторов | 4.255ms | 3.457ms | 4.230ms | 7.331ms | 28.950ms | В пределах с выбросом |
| `groupId` scope, 500 enemy-групп, 5k акторов | 10.185ms | 7.290ms | 27.321ms | 28.604ms | 31.532ms | Выше p95 |

Интерпретация:

- `groupTag` scope не становится дешевым автоматически. Он дешевый только тогда, когда тег соответствует небольшому рабочему набору.
- `groupId` scope лучше подходит для игровых обновлений каждый кадр, потому что позволяет явно контролировать число акторов, получающих событие.
- Один кадр не должен затрагивать тысячи групп акторов, если в этом же кадре игре нужны рендеринг, физика, AI, сеть и browser/runtime overhead.

### Приближенный игровой цикл

Дополнительный benchmark моделирует один полный игровой `FRAME` поверх текущего публичного API:

- `240` domain-машин как системный слой кадра: physics/render/audio/AI/input/network-like counters.
- `12000` акторов в `3100` группах:
  - enemies: `600` групп x `10` = `6000`;
  - projectiles: `1000` групп x `2` = `2000`;
  - props/objects: `500` групп x `4` = `2000`;
  - effects: `1000` групп x `2` = `2000`.
- Один `FRAME` доставляется `unscoped`: domain-системы обновляют frame counters, actor reducers обновляют позицию, ttl/hp и `renderVersion`.
- React render, canvas/WebGL draw calls, collision broadphase, pathfinding и DOM work в замер не входят.

Команда:

```bash
NODE_ENV=production npx vitest bench tests/bench/hot-path.bench.ts --run
```

Результат замера от 2026-04-29:

| Сценарий | Hz | Среднее | Минимум | Максимум | p99 | Сэмплы | Статус 16ms |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| `240` game systems + `12000` actors / full frame | 123.21 | 8.1161ms | 7.2514ms | 11.4493ms | 11.4493ms | 62 | В пределах для FSM-слоя |

Интерпретация: текущий FSM hot path справляется с `~12k` простых actor updates и сотнями domain-систем в пределах `16ms` на этой машине, но такой кадр уже занимает около половины бюджета. Для реальной игры с рендерингом, физикой, AI и browser overhead комфортнее держать per-frame FSM работу ниже `4-8ms` или тикать мир чанками/видимыми группами.

## Ограничения

Этот отчет — baseline, а не финальная гарантия емкости.

- Он не включает React render, invalidation селекторов, DOM work, игровой рендеринг, физику, pathfinding, collision checks, работу с ассетами и стоимость browser compositor.
- Node.js на macOS не полностью совпадает с runtime-профилем browser game loop.
- Результаты включают обычное поведение JavaScript-аллокаций из-за immutable reducer returns.
- Измеряемый путь использует `ts-node/esm`; production-пакет `dist` может немного отличаться после bundling/minification.
- `p99` и `max` чувствительны к GC, планировщику ОС, фоновой нагрузке и thermal/power state CPU.
- Benchmark reducer-ы намеренно простые. Реальные reducer-ы с AI/pathfinding/physics логикой уменьшат безопасный actor budget.

## Рекомендуемый следующий анализ

1. Сделать воспроизводимый benchmark-скрипт, который пишет JSON-результат для сравнения в CI.
2. Добавить browser-based benchmark coverage для тех же сценариев, желательно с Chrome Performance traces.
3. Разделить время actor hot path на routing, выполнение reducer, клонирование state record, sidecar commit и subscriber notification.
4. Сравнить текущие immutable actor record writes с альтернативными внутренними storage layout.
5. Добавить сценарии с no-op actor reducer-ами и более тяжелыми reducer-ами, чтобы отделить framework overhead от user reducer cost.
6. Добавить примеры chunked game loop: обновлять N групп за кадр и ротировать регионы мира по кадрам.
7. Отдельно измерить React-интеграцию: количество selector, rerender count и provider notification overhead.

## Рабочая рекомендация по бюджету

Для game-like 60fps использования на этой машине и текущей реализации:

- Считать `~5k` акторов, получающих один `unscoped` тик, верхней практической baseline-границей только для FSM-слоя.
- Считать `~1k` акторов на один `groupId` scoped тик комфортным диапазоном для FSM-слоя.
- Держать широкие `groupTag` scoped тики ниже нескольких тысяч акторов, иначе ожидать давление на `p95`.
- Оставлять минимум 50-70% от `16ms` кадра на рендеринг и реальную игровую логику; на практике целиться в `MachineManager.transition()` ниже `4-6ms` для per-frame тика.
