# @lite-fsm/persist

`@lite-fsm/persist` — это отдельный модуль, который сохраняет состояние `MachineManager` в любое хранилище и восстанавливает его обратно. Внутри он опирается на методы менеджера: `dehydrate()` для выгрузки снимка, `hydrate()` для применения и `onTransition()` для отслеживания изменений.

Подключайте этот модуль, когда нужно:

- сохранить состояние между перезагрузками страницы;
- синхронизировать состояние между вкладками одного браузера;
- восстановить состояние из произвольного хранилища: `localStorage`, `IndexedDB`, серверной сессии, файла и т.п.

Конфликты при объединении входящего снимка с текущим состоянием разрешают обработчики `hydrate` в ваших машинах — модуль `persist` сам по себе их не разрешает.

## Импорт

`persist` находится в отдельной точке входа, чтобы его не приходилось загружать, когда он не нужен. React-хуки лежат в `@lite-fsm/persist/react` отдельно — так `@lite-fsm/react` остаётся независимым от `persist` и не тянет его в bundle, если хуки не используются.

```ts showLineNumbers copy
import { persistManager, createJsonStorage } from "@lite-fsm/persist";
import type { PersistController, PersistStorage, PersistStatus } from "@lite-fsm/persist";

import { usePersistStatus, useIsPersistRestoring } from "@lite-fsm/persist/react";
```

## Базовый пример

Базовый сценарий: создать менеджер, обернуть его в `persistManager` и передать получившийся контроллер в `FSMContextProvider`. Провайдер сам запустит контроллер при монтировании компонента и остановит при размонтировании.

```tsx showLineNumbers copy
import { MachineManager } from "@lite-fsm/core";
import { immerMiddleware } from "@lite-fsm/middleware/immer";
import { createJsonStorage, persistManager } from "@lite-fsm/persist";
import { FSMContextProvider } from "@lite-fsm/react";

import { counter } from "./machines/counter";

const machines = { counter };
type Store = typeof machines;

const manager = MachineManager<Store>(machines, {
  middleware: [immerMiddleware],
  schemaVersion: 1,
});

const persist = persistManager(manager, {
  storage: createJsonStorage<Store>({
    key: "app:state:v1",
    storage: window.localStorage,
  }),
  storageVersion: 1,
  machines: ["counter"],
  throttleMs: 500,
  onError: console.error,
});

export function App() {
  return (
    <FSMContextProvider machineManager={manager} persist={persist}>
      <Counter />
    </FSMContextProvider>
  );
}
```

`persistManager` возвращает контроллер, но ничего не запускает. Подписки и чтение хранилища начнутся только после вызова `start()` — вручную или через `FSMContextProvider`.

## Контроллер `persistManager`

### Методы

| Метод                 | Что делает                                                                                                                            |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `start()`             | Запускает работу контроллера: подписывается на изменения менеджера и хранилища, читает сохранённую запись. Возвращает функцию остановки. |
| `restore()`           | Читает запись и применяет её к менеджеру через `manager.hydrate()`. Возвращает финальный `PersistStatus`.                              |
| `save()`              | Сразу записывает текущий снимок (`manager.dehydrate({ machines })`) в хранилище.                                                      |
| `flush()`             | Если есть отложенное сохранение из-за `throttleMs`, дописывает его сейчас. Иначе дожидается уже идущей записи.                         |
| `clear()`             | Отменяет отложенное сохранение, удаляет запись из хранилища и сбрасывает статус в `{ phase: "ready", restored: false }`.              |
| `getStatus()`         | Возвращает текущий `PersistStatus`.                                                                                                   |
| `subscribeStatus(cb)` | Подписывает на смену статуса. Возвращает функцию отписки.                                                                              |

`start()` можно вызывать несколько раз — внутри есть счётчик активных подключений, и контроллер реально остановится только после последнего `stop()`. Это безопасно для React StrictMode и удобно, когда один контроллер подключён сразу к нескольким провайдерам.

При прямом вызове `restore`, `save`, `flush` и `clear` ошибки пробрасываются наружу — их можно поймать через `try/catch`. Фоновые операции, запущенные внутри `start()`, ошибки наружу не пробрасывают: иначе получились бы необработанные промисы. Они всё равно сообщают об ошибке через `onError` и переводят контроллер в фазу `"error"`.

### `PersistStatus`

Текущая стадия работы контроллера. Удобно показывать пользователю и логировать.

```ts showLineNumbers copy
type PersistStatus =
  | { phase: "idle" }
  | { phase: "restoring" }
  | { phase: "ready"; restored: boolean }
  | { phase: "error"; error: unknown };
```

| Фаза          | Когда выставляется                                                                                                       |
| ------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `"idle"`      | До первого `start()` и после последнего `stop()`.                                                                       |
| `"restoring"` | Идёт восстановление: чтение из хранилища, миграция, применение `manager.hydrate()`.                                      |
| `"ready"`     | Восстановление завершилось. `restored: true` — снимок применён к менеджеру; `false` — записи в хранилище не было.        |
| `"error"`     | Ошибка в `restore`, `save` или `clear`. После следующего успешного восстановления статус снова станет `"ready"`.         |

## Опции `persistManager`

| Опция               | Назначение                                                                                                                                  |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `storage`           | Адаптер хранилища `{ get, set, remove, subscribe? }`. Может быть синхронным или асинхронным.                                                |
| `machines?`         | Какие машины попадают в снимок. Тот же набор ключей, что и у `manager.dehydrate({ machines })`.                                              |
| `strategy?`         | Стратегия применения снимка через `manager.hydrate(snapshot, { strategy })`. По умолчанию `"merge"`.                                        |
| `storageVersion?`   | Версия записи в хранилище. Если она не совпадает с сохранённой и нет `migrate`, запись удаляется.                                           |
| `maxAge?`           | Срок жизни записи в миллисекундах. После истечения запись удаляется и не применяется.                                                       |
| `throttleMs?`       | Минимальный интервал между записями в хранилище. По умолчанию `0` — записывать сразу. Для браузерного хранилища обычно ставят `300–1000`.   |
| `shouldSave?`       | Фильтр переходов перед сохранением. Возвращает `true`, если переход должен попасть в хранилище.                                              |
| `migrate?`          | Преобразование старой записи в текущий формат снимка. Возврат `undefined` приводит к удалению записи.                                        |
| `onRestoreSettled?` | Колбэк итога восстановления. Вызывается на ready или error. `clear()` его не вызывает.                                                       |
| `onError?`          | Колбэк ошибок: `(err, "restore" \| "save" \| "clear") => void`. Удобно подключить к логгеру.                                                |

### Что попадает в хранилище: `machines` и `shouldSave`

Контроллер фильтрует данные двумя независимыми способами.

`machines` сужает _что именно_ попадает в снимок — это те же ключи, что и у `manager.dehydrate({ machines })`. Так удобно исключить из хранилища ту часть состояния, которая не должна переживать перезагрузку: черновики форм, временное состояние интерфейса, акторы запросов.

`shouldSave` фильтрует _когда именно_ записывать снимок — отдельно для каждого перехода. Это второй уровень фильтра, он работает поверх `machines`. Например, в чате имеет смысл сохранять только подтверждённые сообщения и очистку истории, но не каждое нажатие клавиши:

```ts showLineNumbers copy
const persist = persistManager(manager, {
  storage,
  storageVersion: 1,
  machines: ["chatThread"],
  throttleMs: 250,
  shouldSave: ({ action }) =>
    action.type === "MESSAGE_SENT" || action.type === "HISTORY_CLEARED",
});
```

Служебный переход `@@lite-fsm/HYDRATE`, который менеджер генерирует при `hydrate()`, в `shouldSave` не передаётся и сам по себе сохранение не запускает. Поэтому при синхронизации между вкладками не возникает повторных записей в хранилище.

### `throttleMs`: объединение нескольких сохранений в одно

`throttleMs` задаёт минимальный интервал между записями в хранилище. Когда несколько переходов происходят подряд, контроллер не пишет каждый из них отдельно, а откладывает запись на `throttleMs` миллисекунд и потом сохраняет только итоговое состояние.

Зачем это нужно:

- браузерный `localStorage.setItem` синхронный и блокирует основной поток — частые вызовы заметно тормозят UI;
- у `IndexedDB` и сетевых запросов есть стоимость на каждое обращение, нет смысла отправлять каждое промежуточное состояние;
- при печати в поле ввода или быстром перетаскивании контроллер всё равно сохранит только финальное состояние.

`throttleMs: 0` (по умолчанию) — записывать сразу. Для `localStorage` обычно достаточно `300–1000`. `flush()` принудительно дописывает отложенную запись — это полезно перед `unload` страницы.

## Хранилище

`PersistStorage<S>` — это минимальный интерфейс, который должен реализовать любой адаптер. Контроллер не знает про сериализацию и транспорт: он просто читает, пишет и удаляет записи.

```ts showLineNumbers copy
type PersistStorage<S extends MachineStore> = {
  get(): MaybePromise<PersistedRecord<S> | undefined>;
  set(record: PersistedRecord<S>): MaybePromise<void>;
  remove(): MaybePromise<void>;
  subscribe?(cb: () => void): () => void;
};
```

Запись `PersistedRecord<S>` — это обёртка над `MachineManagerSnapshot<S>` с двумя дополнительными полями метаданных:

```ts showLineNumbers copy
type PersistedRecord<S extends MachineStore> = {
  timestamp: number;
  storageVersion?: string | number;
  snapshot: MachineManagerSnapshot<S>;
};
```

`timestamp` нужен для проверки `maxAge`, `storageVersion` — для миграций.

### `createJsonStorage`

Готовый адаптер для любого синхронного хранилища с интерфейсом `localStorage` (`{ getItem, setItem, removeItem }`). Сериализует запись в JSON через `JSON.stringify` и читает обратно через `JSON.parse`.

```ts showLineNumbers copy
import { createJsonStorage } from "@lite-fsm/persist";

const storage = createJsonStorage<Store>({
  key: "app:state:v1",
  storage: window.localStorage,
});
```

`createJsonStorage` не реализует `subscribe` — у самого `localStorage` нет внутренней подписки. Если нужно реагировать на изменения от других вкладок, дополните адаптер вручную (см. ниже).

### Синхронизация между вкладками

Браузерное событие `storage` срабатывает у других вкладок при изменении `localStorage`. В адаптере достаточно подписаться на это событие и вызвать колбэк — контроллер сам запустит фоновое восстановление.

```ts showLineNumbers copy
const STORAGE_KEY = "app:state:v1";

const adapter: PersistStorage<Store> = {
  ...createJsonStorage<Store>({ key: STORAGE_KEY, storage: window.localStorage }),
  subscribe: (cb) => {
    const handle = (event: StorageEvent) => {
      if (event.storageArea !== window.localStorage) return;
      if (event.key !== null && event.key !== STORAGE_KEY) return;
      cb();
    };

    window.addEventListener("storage", handle);
    return () => window.removeEventListener("storage", handle);
  },
};
```

Если восстановление пришло из `subscribe`, контроллер не пишет данные обратно в хранилище — иначе соседние вкладки получили бы эхо. Запись произойдёт только если локальное состояние реально изменилось во время этого восстановления. Поэтому синхронизация между вкладками не зацикливается.

### Кастомные адаптеры

Тот же контракт подходит для IndexedDB, BroadcastChannel, серверных сессий, файлового хранилища или любого другого транспорта. Главное:

- из `get` возвращать `PersistedRecord<S>` или `undefined`, если записи нет;
- идемпотентно реагировать на повторные вызовы `set` и `remove`;
- если есть внешние изменения — реализовать `subscribe`.

Простой in-memory адаптер удобен для тестов:

```ts showLineNumbers copy
const memoryAdapter = (): PersistStorage<Store> => {
  let record: PersistedRecord<Store> | undefined;
  const listeners = new Set<() => void>();

  return {
    get: () => record,
    set: (next) => {
      record = next;
      for (const cb of listeners) cb();
    },
    remove: () => {
      record = undefined;
      for (const cb of listeners) cb();
    },
    subscribe: (cb) => {
      listeners.add(cb);
      return () => listeners.delete(cb);
    },
  };
};
```

## React-интеграция

### Свойство `persist` у `FSMContextProvider`

`FSMContextProvider` принимает свойство `persist`, которое управляет жизненным циклом контроллера: при монтировании провайдер вызывает `start()`, при размонтировании — возвращённую функцию остановки. Серверный рендер и React-гидратация при этом не блокируются: первое восстановление идёт в фоне после монтирования.

```tsx showLineNumbers copy
import { FSMContextProvider } from "@lite-fsm/react";

<FSMContextProvider machineManager={manager} persist={persist}>
  <App />
</FSMContextProvider>
```

`persist` принимает один контроллер или массив. Так удобно подключить сразу два — например, один пишет в `localStorage`, другой отправляет снимок на сервер:

```tsx showLineNumbers copy
<FSMContextProvider machineManager={manager} persist={[localPersist, remotePersist]}>
  <App />
</FSMContextProvider>
```

Тип свойства — это любой объект с методом `start(): () => void`. Туда можно передать свой контроллер с таким же интерфейсом или мок в тестах.

### `usePersistStatus`, `useIsPersistRestoring`

Эти хуки подписываются на изменение статуса контроллера и возвращают актуальное значение. Удобно показывать индикатор синхронизации, ошибку или скрывать UI на время восстановления.

```tsx showLineNumbers copy
import { useIsPersistRestoring, usePersistStatus } from "@lite-fsm/persist/react";
import type { PersistController } from "@lite-fsm/persist";

function PersistBadge({ persist }: { persist: PersistController }) {
  const status = usePersistStatus(persist);
  const restoring = useIsPersistRestoring(persist);

  if (restoring) return <span>Загружаем сохранённые данные…</span>;
  if (status.phase === "error") return <span>Не удалось прочитать запись</span>;
  if (status.phase === "ready" && status.restored) return <span>Синхронизировано</span>;

  return <span>Готово</span>;
}
```

`useIsPersistRestoring` — это сокращение для `usePersistStatus(p).phase === "restoring"`.

## Версии и миграции

В библиотеке две независимые версии — у каждой своя роль. Их легко перепутать:

| Поле                                 | За что отвечает                                                                                                |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| `MachineManager({ schemaVersion })`  | Версия модели состояния (`MachineManagerSnapshot`). Несовпадение вызывает `onSchemaVersionMismatch` менеджера. |
| `persistManager({ storageVersion })` | Версия записи в хранилище (`PersistedRecord`). Несовпадение приводит к удалению записи или вызову `migrate`.   |

Как выбрать, что менять:

- меняли структуру состояния машин — поднимите `schemaVersion`;
- меняли формат хранилища (ключи, метаданные, способ сериализации) — поднимите `storageVersion`.

```ts showLineNumbers copy
const persist = persistManager(manager, {
  storage,
  storageVersion: 2,
  migrate: (record) => {
    if (record.storageVersion === 1) return upgradeFromV1(record.snapshot);
    return undefined;
  },
});
```

Поведение при несовпадении версии:

- без `migrate` — запись удаляется, статус становится `{ phase: "ready", restored: false }`;
- с успешным `migrate` — возвращённый снимок применяется, контроллер тут же перезаписывает запись с актуальным `storageVersion`;
- `migrate` вернул `undefined` — запись считается несовместимой и удаляется.

`maxAge` задаёт срок жизни записи в миллисекундах. Просроченные записи удаляются ещё до того, как может вызваться `migrate`.

## Жизненный цикл

`start()` синхронно возвращает функцию остановки, а реальное чтение из хранилища и установка подписок идут в фоне. Поэтому он не задерживает рендер и его безопасно вызывать прямо в `useEffect`.

Что важно понимать про работу контроллера между `start()` и `stop()`:

- если пользовательский переход случился во время восстановления, контроллер не пишет его в хранилище сразу, а помечает «есть несохранённые изменения». После завершения восстановления он сохранит итоговое состояние;
- `restore`, `save`, `flush`, `clear` можно вызывать вручную в любой момент;
- `clear()` отменяет отложенное сохранение, удаляет запись и помечает уже идущие операции как устаревшие — даже если запущенный ранее `save` или `restore` завершится после `clear()`, он уже не перезапишет хранилище.

```ts showLineNumbers copy
const stop = persist.start();

await persist.flush();
await persist.clear();

stop();
```

## Практический пример: чат с синхронизацией между вкладками

В <a href={playgroundExamplePath("persist")} style={{ color: "#2563eb", textDecoration: "underline", textUnderlineOffset: "2px" }}>примере persist в playground</a> история чата сохраняется в `localStorage`, а соседние вкладки получают её через событие `storage`. В хранилище попадает только машина `chatThread`: композер ввода (`chatComposer`) и сессия (`chatSession`) живут только в текущей вкладке.

```ts showLineNumbers copy
const machines = { chatThread, chatComposer, chatSession };
type Store = typeof machines;

const manager = MachineManager<Store>(machines, {
  middleware: [immerMiddleware],
  schemaVersion: 1,
});

manager.transition({
  type: "SESSION_STARTED",
  payload: { peer, openedAt: Date.now() },
});

const persist = persistManager(manager, {
  storage: createLocalStorageAdapter(window.localStorage),
  storageVersion: 1,
  machines: ["chatThread"],
  throttleMs: 250,
  shouldSave: ({ action }) =>
    action.type === "MESSAGE_SENT" || action.type === "HISTORY_CLEARED",
  onError: console.error,
});
```

```tsx showLineNumbers copy
import { FSMContextProvider } from "@lite-fsm/react";
import { useIsPersistRestoring, usePersistStatus } from "@lite-fsm/persist/react";

function ChatApp({ persist }: { persist: PersistController }) {
  const status = usePersistStatus(persist);
  const restoring = useIsPersistRestoring(persist);

  return (
    <section>
      {restoring && <p>Подключаем хранилище…</p>}
      <ChatPanel />
      <ChatComposer />
      {status.phase === "error" && <p>Ошибка хранилища</p>}
    </section>
  );
}

export function App() {
  return (
    <FSMContextProvider machineManager={manager} persist={persist}>
      <ChatApp persist={persist} />
    </FSMContextProvider>
  );
}
```

Откройте демо в нескольких вкладках одновременно: каждая вкладка — свой пользователь, но история сообщений общая. Если очистить её в одной вкладке, остальные тут же подтянут пустой `chatThread`.

См. также страницу [Сохранение состояния](/guide/hydration), где описана базовая модель `dehydrate`/`hydrate`, поверх которой работает `persist`.

## Обработка ошибок

`onRestoreSettled` и `onError` дополняют друг друга:

- `onError` вызывается на каждую ошибку и сообщает её фазу (`restore`, `save`, `clear`). Удобно подключить к логгеру или системе мониторинга;
- `onRestoreSettled` вызывается ровно один раз на каждый цикл восстановления — успешный или нет. Удобно для аналитики и метрик.

```ts showLineNumbers copy
const persist = persistManager(manager, {
  storage,
  storageVersion: 1,
  onRestoreSettled: (result) => {
    if (result.phase === "ready") {
      analytics.track("persist:ready", { restored: result.restored });
      return;
    }
    analytics.track("persist:restore_error", { error: String(result.error) });
  },
  onError: (err, phase) => {
    console.error(`[persist] ${phase}`, err);
  },
});
```

| Источник ошибки                       | Что вызывается                                                                                         |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `restore` (включая невалидную запись) | `onError(err, "restore")`, статус `{ phase: "error" }`, `onRestoreSettled({ phase: "error", error })`. |
| `save`                                | `onError(err, "save")`, статус `{ phase: "error" }`.                                                   |
| `clear`                               | `onError(err, "clear")`. При прямом вызове ошибка пробрасывается дальше.                              |

Если ваш `onError` или `onRestoreSettled` сам бросит исключение, контроллер его подавит — основная логика `persist` продолжит работать.

## Ограничения

- Сериализация и совместимость пользовательских данных в записи остаются за вашим адаптером.
- Служебный переход `@@lite-fsm/HYDRATE` не запускает сохранение.
- `manager.hydrate()` не проходит через middleware и не запускает эффекты — это поведение самого менеджера, и `persist` тут ничего не меняет.
- Запросы к IndexedDB, серверу, файлам и другим асинхронным хранилищам пишутся вами; контроллер только корректно вызывает `get`, `set`, `remove` и реагирует на `subscribe`.
